# Tailwind UI Modernization — Design Spec

**Status:** Approved design, pre-implementation
**Read first:** `AGENTS.md`, `docs/superpowers/specs/2026-07-08-ftle-design.md`

## Motivation

The current UI (public site and admin editor) is functionally complete but visually minimal to the point of being unstyled: system fonts, no color beyond OS default, a 21-line hand-written CSS file (`src/css/site-css.ts`), and a completely unstyled admin editor (raw `<form>`/`<input>` with one inline `style="color:red"`). The goal is a more modern, polished look — refined minimalism, not a heavy visual overhaul — while preserving the project's core constraints.

## Non-negotiable constraints (from `AGENTS.md`)

- Reader-facing pages: **0 bytes of JavaScript, 0 blocking external requests, ≤ 14KB compressed**, enforced by `tests/perf/page-weight.test.ts`. This does not change.
- Admin pages are exempt from the JS/weight budget but **share the same inlined stylesheet** as public pages (one `<style>` block, written once in `layout.ts`).
- No external stylesheet — CSS stays inlined into every page. The Tailwind **CDN `<script>` build is therefore ruled out** (it's JS, and it fetches from an external origin). Only a **build-time compiled, purged** Tailwind output is compatible with this architecture.
- No additional *runtime* dependencies without discussion. Tailwind is added as a **devDependency only** — it runs at build time and contributes nothing to the deployed Worker bundle beyond the static CSS string it generates.

## Scope

Both the public site (`src/routes/public.ts`, `src/layout.ts`) and the admin editor (`src/routes/admin.ts`). `src/routes/rss.ts` emits Atom XML only, with no HTML/CSS surface, so it's untouched.

## Build pipeline

Mirrors the existing KaTeX asset pipeline (`scripts/prepare-katex-assets.mjs` → `src/generated/katex-manifest.ts`) exactly, so the pattern is already familiar to this codebase:

1. **`tailwindcss` added as a devDependency.**
2. **`tailwind.config.mjs`** — `content` globs over `src/**/*.ts` (Tailwind's static analyzer extracts class names via regex from any file matching the glob, including `.ts` files with template-literal HTML). `darkMode: 'media'` — public pages have zero JS, so dark mode can only ever be system-preference-driven (`prefers-color-scheme`), never a click-to-toggle switch. `theme.fontFamily` is overridden to the existing system-font stack (`-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial, sans-serif`) — no webfonts are introduced, preserving the speed baseline described in the 2026-07-08 spec.
3. **`src/css/tailwind-input.css`** — the three `@tailwind` directives (`base`, `components`, `utilities`) plus a minimal `@layer base` for the one non-utility global rule (`color-scheme: light dark` on `html`).
4. **`scripts/build-css.mjs`** — invokes the Tailwind CLI against `tailwind-input.css` with `--minify`, reads the compiled output, and writes it into `src/generated/site-css.ts` as `export const SITE_CSS = "...";` with a `// Generated by scripts/build-css.mjs — do not edit.` banner, matching `katex-manifest.ts`'s convention. `src/generated/` is already gitignored, so this generated file is never committed.
5. **`package.json`** — add `"prepare:css": "node scripts/build-css.mjs"`, and append it to the existing `predev` and `pretest` scripts (same slot `prepare:katex` occupies today).
6. **Delete `src/css/site-css.ts`.** `src/layout.ts` imports `SITE_CSS` from `./generated/site-css` instead of `./css/site-css`.

### Why no new automated test for the build script itself

`scripts/katex-manifest.mjs` (the precedent this mirrors) has no direct unit test either — it's build tooling, not application behavior. The thing that *is* behavior, and *is* already tested, is the final rendered HTML weight: `tests/perf/page-weight.test.ts` measures the gzipped size of full pages including the inlined `<style>` block. That test is the acceptance gate for the Tailwind output — if purged utility CSS pushes a page over 14KB, that's a failing test, fixed by trimming utility variety (per `AGENTS.md`'s "treat a violation exactly like any other failing test — fix the cause, never raise the limit").

## Visual design — refined minimalism

- **Typography**: keep the system-font stack; use Tailwind's type scale for real hierarchy (larger/tighter `h1` for post titles), a comfortable content measure (replacing the flat `max-width: 42rem`), improved line-height for body text.
- **Color**: one muted accent color (indigo/blue family) used sparingly for links and hover states — not backgrounds or large fills. Light/dark neutral palette via `dark:` variants under the `media` strategy, so theme still follows system preference with zero JS, same behavior as today's `color-scheme: light dark`.
- **Spacing/rhythm**: consistent vertical spacing between nav / article / footer via Tailwind's spacing scale, replacing today's ad hoc `margin-bottom` values.
- **Post list**: light visual separation between entries (hairline border or spacing — not full "cards"), post date de-emphasized via muted color + smaller size utilities.
- **Code blocks**: subtle background + border radius, using tokens that respect dark mode (today's `rgba(127,127,127,0.1)` background does not adapt to theme).
- **Admin editor**: styled form — labeled inputs with real spacing, a monospace `<textarea>` for Markdown source, a clearer Preview/Save button pair, error messages restyled from inline `style="color:red"` to a proper alert-style utility combination. No new JS — the existing iframe-based live preview mechanism is unchanged.

## Risks & mitigations

- **Shared stylesheet risk**: admin's utility classes add to the weight of *every* page (including public ones), since both pull from one compiled `SITE_CSS`. Mitigation: keep admin markup's utility usage modest (it's one form and one list); verify against `tests/perf/page-weight.test.ts` as pages are converted, not just at the end.
- **Tailwind purge blind spots**: dynamic class construction (e.g. `` `text-${size}` ``) is invisible to Tailwind's static analyzer and would silently produce unstyled markup. Mitigation: always write full class name literals in the route/layout template strings, never interpolate partial class names.
- **Existing tests asserting on markup**: confirmed no test in `tests/` asserts on layout-level class names like `.post-list`/`.post-date` (only on content-level classes like `class="katex"` and `class="callout"`, which come from post *content*, not layout chrome, and are untouched by this change).

## Testing / verification plan

- `npm test` must stay green throughout, including `tests/perf/page-weight.test.ts` (the real gate for CSS weight) and `tests/integration/public-routes.test.ts` / admin route tests (structural HTML expectations, if any — verified none assert on layout class names).
- `npm run typecheck` green.
- Manual verification: `npm run dev`, visually check index/post/tag pages and the admin new/edit forms in both light and dark OS theme.

## Out of scope

- No new JS anywhere on public pages (unchanged non-negotiable).
- No webfonts.
- No visual redesign of the Markdown/KaTeX rendering output itself (headings inside post content, KaTeX styling) — only the site chrome (layout, nav, post list, admin forms).
