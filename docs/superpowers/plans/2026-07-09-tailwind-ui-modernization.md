# Tailwind UI Modernization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Modernize the visual design of the public site and admin editor (refined minimalism: real typography scale, subtle accent color, dark mode, polished admin form) using a build-time-compiled, purged Tailwind CSS v4 stylesheet, while keeping the zero-JS / ≤14KB-gzipped public-page budget intact.

**Architecture:** Tailwind v4 (CSS-first config, no `tailwind.config.js`) compiles `src/css/tailwind-input.css` into a single minified stylesheet via the standalone `@tailwindcss/cli` package. A new `scripts/build-css.mjs` — mirroring the existing `scripts/prepare-katex-assets.mjs` pattern — writes that output into `src/generated/site-css.ts` (gitignored, regenerated by `predev`/`pretest`). `src/layout.ts` inlines it into the same `<style>` block it uses today; only the source of that string changes. Route templates (`src/layout.ts`, `src/routes/public.ts`, `src/routes/admin.ts`) are then updated in place to use Tailwind utility classes instead of the old hand-written CSS selectors.

**Tech Stack:** Tailwind CSS v4.3.2 (`@tailwindcss/cli`, `@tailwindcss/typography`), Hono, TypeScript, Vitest + `@cloudflare/vitest-pool-workers`.

**Read first:** `AGENTS.md`, `docs/superpowers/specs/2026-07-08-ftle-design.md`, `docs/superpowers/specs/2026-07-09-tailwind-ui-modernization-design.md` (this plan implements that spec).

## Global Constraints

- Reader-facing pages: 0 bytes of JavaScript, 0 blocking external requests, ≤ 14KB compressed. Enforced by `tests/perf/page-weight.test.ts` — never raise this limit; fix the cause if a task pushes a page over it.
- Admin pages are exempt from the JS/weight budget, but share the same inlined `<style>` block as public pages — admin's utility classes add to every page's weight, so keep admin markup's utility variety modest.
- No external stylesheet — CSS stays inlined into every page (the versioned KaTeX CSS on `has_math` pages is the sole existing exception; unchanged by this plan).
- Tailwind is a devDependency only — no new runtime dependency is added to the deployed Worker.
- Never write a dynamically-constructed class name (e.g. `` `text-${size}` ``) anywhere in route/layout template strings — Tailwind's static analyzer can't see it, and it silently produces unstyled markup. Always write full class-name literals.
- TDD is required for behavior changes (AGENTS.md). Tasks in this plan that are pure presentational (CSS class additions, no new logic branch) rely on the existing test suite as the regression gate rather than a new failing test, per the spec's documented rationale (mirrors `scripts/katex-manifest.mjs`, which has no dedicated unit test either). Task 4 is the one exception with a genuine test change: two assertions in `tests/integration/admin-editor.test.ts` pin an attribute-free `<h1>` and must be loosened to tolerate a `class` attribute.
- Commit messages: imperative mood, one logical change per commit.

```css
@import "tailwindcss";
@plugin "@tailwindcss/typography";
@source "../**/*.ts";

@theme {
  --font-sans: -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial, sans-serif;
}

@layer base {
  html {
    color-scheme: light dark;
  }
}
```

Notes for the implementer:
- `@source "../**/*.ts"` is relative to this file's directory (`src/css/`), so it resolves to `src/**/*.ts` — every route/layout template in the project.
- No `darkMode` setting is needed: Tailwind v4's `dark:` variant defaults to `@media (prefers-color-scheme: dark)` unless a custom `dark` variant is declared (which this file does not do). That is the system-preference-driven, zero-JS behavior this project needs.
- `@plugin "@tailwindcss/typography"` registers the `prose`/`prose-invert` utilities used in Task 3 for post content.

- [ ] **Step 3: Create the CSS build script**

Create `scripts/build-css.mjs`:

```js
import { execFileSync } from "node:child_process";
import { mkdirSync, readFileSync, rmSync, writeFileSync } from "node:fs";
import { tmpdir } from "node:os";
import path from "node:path";
import { fileURLToPath } from "node:url";

const root = path.dirname(fileURLToPath(import.meta.url)) + "/..";
const inputPath = path.join(root, "src/css/tailwind-input.css");
const generatedDir = path.join(root, "src/generated");
const tailwindBin = path.join(root, "node_modules/.bin/tailwindcss");
const tmpOutput = path.join(tmpdir(), `ftle-tailwind-${process.pid}.css`);

mkdirSync(generatedDir, { recursive: true });

execFileSync(tailwindBin, ["-i", inputPath, "-o", tmpOutput, "--minify"], {
  cwd: root,
  stdio: "inherit",
});

const css = readFileSync(tmpOutput, "utf8").trim();
rmSync(tmpOutput, { force: true });

writeFileSync(
  path.join(generatedDir, "site-css.ts"),
  `// Generated by scripts/build-css.mjs — do not edit.\nexport const SITE_CSS = ${JSON.stringify(css)};\n`,
);

console.log(`prepare:css — wrote src/generated/site-css.ts (${css.length} bytes)`);
```

This mirrors `scripts/prepare-katex-assets.mjs`'s structure (resolve `root` from `import.meta.url`, write into `src/generated/`, log a one-line summary).
